# Open Knowledge Format (OKF) Specification

**Version 1.0 — Final Standard**

The **Open Knowledge Format (OKF)** is an open, vendor-neutral, tool-agnostic standard for representing software architecture, codebase metadata, domain context, API contracts, and operational knowledge. 

It is designed to be authored by engineers, generated by tools, versioned in Git, and consumed seamlessly by **every coding agent** (e.g., Antigravity, Claude Code, Cursor, Copilot, Windsurf, and custom LLM agents).

---

## 1. Vision & Core Philosophy

As AI coding agents become primary software development partners, managing project context, architecture constraints, and domain knowledge is critical. Proprietary cloud portals, vendor-locked metadata registries, and heavy database platforms introduce friction, vendor lock-in, and agent isolation.

OKF solves this by grounding knowledge representation on established, universal developer primitives:
- **Plain-text Markdown (`.md`)** for human readability and LLM comprehension.
- **Structured YAML Frontmatter** for deterministic parsing, querying, and indexing.
- **Git Repository Storage** for versioning, diffs, code review, and history.
- **Graph-Shaped Hyperlinks** for expressive concept inter-relationships.

### Goals
1. **Universal Agent Interoperability**: Every coding agent can read, search, traverse, and write OKF bundles using simple file I/O or standard tools (MCP/CLI).
2. **Zero Vendor Lock-in**: No proprietary SDKs, cloud accounts, or SaaS infrastructure required.
3. **Progressive Disclosure**: Agents can read lightweight index manifests (`index.md`) to discover relevant concepts without overwhelming context windows.
4. **Bidirectional Human/Agent Collaboration**: Humans and AI agents collaborate on the exact same markdown artifacts in Git.

## 2. Bundle Structure & Hierarchy

An OKF **Knowledge Bundle** is a self-contained directory tree of markdown files. A project MAY place its bundle in `.okf/`, `knowledge/`, or at the repository root.

```
my-project/
├── .okf/                       # Knowledge bundle directory (or knowledge/)
│   ├── config.yaml            # Optional bundle configuration
│   ├── index.md               # Root manifest & progressive disclosure entry point
│   ├── log.md                 # Chronological change log
│   ├── architecture/          # Grouping: Architecture & System Design
│   │   ├── index.md
│   │   ├── system-overview.md
│   │   └── authentication.md
│   ├── services/              # Grouping: Microservices / Backend Components
│   │   ├── index.md
│   │   └── user-service.md
│   ├── api/                   # Grouping: API Contracts & Endpoints
│   │   ├── index.md
│   │   └── auth-endpoints.md
│   ├── database/              # Grouping: Schemas, Data Models & Tables
│   │   ├── index.md
│   │   └── users-table.md
│   └── rules/                 # Grouping: Coding Guidelines & Conventions
│       ├── index.md
│       └── security-rules.md
```

### 2.1 Reserved Filenames

| Filename     | Scope                 | Purpose |
|--------------|-----------------------|---------|
| `config.yaml` | Bundle Root           | Declares bundle metadata, name, version, and default settings. |
| `index.md`   | Any Directory         | Progressive disclosure directory index. Enumerates child concepts. |
| `log.md`     | Any Directory         | Chronological history of additions, updates, and deprecations. |

All other `.md` files are **Concept Documents**.

## 3. Concept Documents

Every concept document is a UTF-8 encoded markdown file containing two distinct sections:
1. **YAML Frontmatter**: Delimited by `---` lines at the top of the file.
2. **Markdown Body**: Freeform, structured markdown text.

### 3.1 YAML Frontmatter Schema

```yaml
---
type: Service                         # REQUIRED: High-level concept classification
title: User Service                   # RECOMMENDED: Display title
description: Handles user authentication, profile management, and sessions. # RECOMMENDED: 1-line summary
resource: src/services/user/         # OPTIONAL: Codebase file path, URL, or canonical asset URI
tags: [backend, auth, security]       # OPTIONAL: Cross-cutting category tags
status: active                        # OPTIONAL: active | draft | deprecated
owner: team-security                  # OPTIONAL: Team or engineer handle
version: 1.2.0                        # OPTIONAL: Semantic version
timestamp: 2026-07-22T00:00:00Z       # OPTIONAL: ISO 8601 modification timestamp
aliases: [AuthService, UserMgr]       # OPTIONAL: Alternative lookup names
---
```

#### Field Specifications:
- `type` (**REQUIRED**): Short string identifying the category.
- `title` (*Recommended*): Human-readable name. If omitted, consumers derive the title from the filename or `# Heading 1`.
- `description` (*Recommended*): Single-sentence summary. Crucial for `index.md` previews and agent context snippets.
- `resource` (*Optional*): Path to source code (`src/api/auth.go`), schema file, or external reference URL.
- `tags` (*Optional*): Array of string keywords for faceted search.
- `status` (*Optional*): Lifecycle state (`active`, `draft`, `deprecated`).
- `aliases` (*Optional*): List of alternative identifiers for agent alias matching.

#### Standard Concept Types:
OKF does not restrict concept types, but standardizes the following common types for software engineering:

- **Architecture**: System architecture decisions, design patterns, topology.
- **Service**: Microservice, server process, or distinct software module.
- **API**: OpenAPI endpoint, REST resource, GraphQL schema, RPC interface.
- **Database / Table**: Relational table, NoSQL collection, or data model schema.
- **Codebase**: Package, class, module, function, or framework component.
- **Rule**: Coding standard, linting rule, security directive, architectural constraint.
- **Runbook**: Incident response steps, deployment guide, setup instructions.
- **Concept**: Domain term, business logic rule, or background context.

---

### 3.2 Markdown Body Conventions

The body contains freeform markdown. Producers SHOULD use clear section headers (`#`, `##`) to enable section-level retrieval by agents:

| Recommended Section | Purpose |
|---------------------|---------|
| `# Overview`        | Core purpose, context, and high-level architectural responsibility. |
| `# Interface` / `# Schema` | Input/output formats, column types, or API signatures. |
| `# Dependencies`    | Services, tables, or modules required by this concept. |
| `# Usage`           | Code snippets, example queries, or request/response payloads. |
| `# Rules & Constraints` | Specific guidelines, invariants, or security requirements. |
| `# Citations`       | Sourced links, PRs, ADRs, or reference documentation. |

## 4. Graph Model & Cross-Linking

OKF concepts form a **directed graph**. Relationships between concepts are expressed via standard Markdown links:

### 4.1 Link Types

1. **Bundle-Absolute Links** (Recommended):
   Starts with `/`, relative to the bundle root:
   ```markdown
   Depends on the [User Table](/database/users-table.md) for session storage.
   ```
2. **Relative Links**:
   Standard relative paths within subdirectories:
   ```markdown
   See neighboring [Security Rules](./security-rules.md).
   ```

### 4.2 Graph Navigation & Backlinks
Consumers parsing an OKF bundle extract directed edges `(Source Concept -> Target Concept)` from these links.
- **Forward Links**: Explicit dependencies and associations.
- **Backlinks ("Cited By")**: Concepts referencing the target concept.

---

## 5. Progressive Disclosure (`index.md`)

To prevent filling LLM context windows with unnecessary data, OKF uses **Progressive Disclosure**:

An `index.md` file in any directory summarizes the directory contents:

```markdown
# Services Directory

- [User Service](user-service.md) - Handles user authentication, profile management, and sessions.
- [Payment Service](payment-service.md) - Processes credit card and subscription billing.

# Subdirectories

- [API Endpoints](api/index.md) - OpenAPI endpoints and routes.
```

When an agent needs knowledge, it reads `index.md` first, identifies relevant concept paths, and requests only the specific `.md` files needed for the task.

## 6. Coding Agent Integration Protocol

OKF specifies how coding agents (Antigravity, Claude Code, Cursor, Copilot, etc.) interact with knowledge bundles:

### 6.1 Agent System Prompt Rule
Every coding agent working on an OKF-enabled repository MUST be instructed to:
1. Locate `.okf/` or `knowledge/` at repository initialization.
2. Read `.okf/index.md` to understand system architecture and rules before executing code edits.
3. Obey directives stored under `type: Rule` documents.
4. Update or add concept documents when adding new services, tables, or major architectural changes.

### 6.2 Agent Tooling & MCP Interface
Coding agents MAY query OKF bundles via the `okf` CLI or Model Context Protocol (MCP) server:
- `okf_list_concepts`: Returns all concepts with frontmatter metadata.
- `okf_get_concept`: Fetches concept body and forward/backlinks.
- `okf_search_concepts`: Semantic/text search across tags, titles, and bodies.
- `okf_get_context`: Formats a concise bundle summary for prompt injection.
- `okf_validate`: Validates bundle integrity, links, and schema compliance.
- `okf_remember`: Writes a new free-form concept (rule, insight, runbook step, etc.), satisfying the "add concept documents" duty in 6.1 without requiring a mechanical harvest.

---

## 7. Conformance Checklist

A bundle is **OKF v1.0 Conformant** if:
- [x] All concept files contain parseable YAML frontmatter with a `type` field.
- [x] Internal links reference valid bundle paths or existing concepts.
- [x] Reserved filenames (`index.md`, `log.md`) conform to layout requirements.
- [x] Files are UTF-8 encoded with standard Unix newline (`\n`) line endings.
